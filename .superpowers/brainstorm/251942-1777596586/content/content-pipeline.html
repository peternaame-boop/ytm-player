Key decisions

  • Pin to release tag, not master. A docs edit upstream doesn't silently change prod. PINNED_TAG=v1.9.2 in env. Rebuild on tag push bumps it automatically.
  • Local dev cache. First build fetches from GitHub; subsequent builds read from .cache/ytm-docs/. Cleared by CI on every prod build.
  • Markdown stack: remark-parse → remark-gfm → remark-rehype → rehype-shiki → rehype-stringify. Shiki for syntax highlight (theme: ynh-coral matching the site).
  • CHANGELOG splitter. Custom node script parses ### vX.Y.Z (YYYY-MM-DD) headings, emits one Astro page per version. Same regex you already use in publish.yml to extract release notes.
  • No CMS, no frontmatter on the source side. The .md files in the public repo stay raw markdown; all metadata (route, title, OG image, position in tab order) lives in a small site/content-config.ts in the website repo.

Failure modes (handled)

  • GitHub down at build time. Build script falls back to last cached fetch under .cache/ytm-docs/ (committed in CI for this exact case) and emits a warning, doesn't fail.
  • Tag doesn't exist. Build fails loudly with the missing tag name (better than silently shipping stale content).
  • Malformed CHANGELOG section. Splitter logs the bad heading, skips that version, ships the rest.
  • Doc upstream renamed. Build fails with "expected docs/keys.md, got 404." Forces an explicit fix in the website repo's content-config.
  • CF Pages build timeout. Static site, 52 pages, no image processing → typical build ~30-45s. Well under any limit.
One thing worth noting: The website repo never directly imports anything from the ytm-player repo at runtime — only at build time. The deployed site is fully self-contained static HTML. The ytm-player repo could be deleted tomorrow and ytm-player.com would keep working until the next rebuild.
